谷歌如何写设计文档

Posted on 2024-05-08 Edited on 2024-05-11 In 技术 Views: Word count in article: 3.9k Reading time ≈ 14 mins.

原文连接: https://www.industrialempathy.com/posts/design-docs-at-google/
Google 软件工程文化的关键要素之一是使用设计文档来定义软件设计。这些是软件系统或应用程序的主要作者在开始编码项目之前创建的相对非正式的文档。设计文档记录了高层实施策略和关键设计决策，重点是在这些决策过程中考虑的权衡。

作为软件工程师，我们的工作本身不是编写代码，而是解决问题。非结构化文本（例如设计文档的形式）可能是在项目生命周期早期解决问题的更好的工具，因为它可能更简洁、更容易理解，并且在比代码更高的级别上传达问题和解决方案。
除了软件设计的原始文档之外，设计文档在软件开发生命周期中还履行以下功能：

  • 在进行更改时及早发现设计问题的成本仍然较低。
  • 在组织中围绕设计达成共识。
  • 确保考虑跨领域问题。
  • 将高级工程师的知识扩展到组织中。
  • 形成围绕设计决策的组织记忆的基础。
  • 充当软件设计者的技术组合中的摘要工件。

设计文档剖析

设计文档是非正式文档，因此其内容不遵循严格的指南。规则1 是：以对特定项目最有意义的任何形式编写它们。
话虽如此，某种结构已经证明自己确实有用。

背景和范围

本节为读者提供了对正在构建的新系统的情况,以及实际正在构建的内容的非常粗略的概述。这不是需求文档。保持简洁！目标是让读者快速了解情况，但可以假设一些先前的知识，并可以链接到详细信息。本节应完全关注客观背景事实。

目标和非目标

系统目标的简短列表，有时更重要的是非目标是什么。请注意，非目标并不是像“系统不应该崩溃”这样被否定的目标，而是那些可以合理地作为目标但被明确选择不作为目标的东西。一个很好的例子是“ACID 合规性”；在设计数据库时，您肯定想知道这是目标还是非目标。如果它不是目标，您仍然可以选择提供它的解决方案，只要它不会带来妨碍实现目标的权衡。

实际设计

本节应从概述开始，然后详细介绍。

设计文档是记录您在设计软件时所做的权衡的地方。专注于这些权衡，以生成具有长期价值的有用文档。也就是说，在给定背景（事实）目标和非目标（要求）的情况下，设计文档是建议的解决方案，并展示为什么这个解决方案最能满足这些目标的地方。
通过更正式的媒介编写文档的目的是提供灵活性，以适当的方式表达手头的问题。因此，对于如何实际描述设计没有明确的指导。
话虽如此，一些最佳实践和重复主题已经出现，它们对大部分设计文档都有意义：

系统上下文图

在许多文档中，系统上下文图可能非常有用。这样的图表将系统显示为更大技术view的一部分，并允许读者根据他们已经熟悉的环境将新设计置于上下文中。

API

如果正在设计的系统公开了一个 API，那么勾画出该 API 通常是一个好主意。然而，在大多数情况下，人们应该抵制住将正式接口或数据定义复制粘贴到文档中的诱惑，因为这些定义通常很冗长，包含不必要的细节，并且很快就会过时。相反，应关注与设计及其权衡相关的部分。

数据存储

存储数据的系统可能应该讨论这种情况如何以及以何种大致形式的发生。与 API 的建议类似，出于同样的原因，应避免复制粘贴完整的架构定义。相反，应关注与设计及其权衡相关的部分。

代码和伪代码

设计文档不应包含代码或伪代码，除非描述新颖的算法。根据需要，链接到显示设计可实施性的原型。

约束程度

影响软件设计以及设计文档的主要因素之一是解决方案空间的约束程度。

极端的一端是“绿地软件项目”，我们所知道的只是目标，解决方案可以是任何最有意义的。这样的文档可能范围广泛，但它还需要快速定义一组规则，以允许放大一组可管理的解决方案。

另一方面，系统中可能的解决方案已经被很好地定义，但如何将它们组合起来以实现目标却并不明显。这可能是一个难以更改的遗留系统，并且不是为执行您想要的操作而设计的，或者是一个需要在主机编程语言的约束下运行的库设计。

在这种情况下，您可能能够相对轻松地列举出所有可以做的事情，但您需要创造性地将这些事情组合在一起才能实现目标。可能有多种解决方案，但没有一个是真正出色的，因此这样的文档应侧重于在考虑到所有已确定的权衡后选择最佳方法。

考虑的替代方案

本节列出了可以合理实现类似结果的替代设计。重点应该放在每个设计所做的权衡以及这些权衡如何导致选择作为文档主要主题的设计的决定。

虽然简洁地介绍最终没有被选择的解决方案是可以的，但本节是最重要的部分之一，因为它非常明确地说明了为什么所选的解决方案是给定项目目标的最佳解决方案，以及读者可能会如何选择其他解决方案想知道，考虑到目标，引入不太理想的权衡。

跨领域关注

在这里，您的组织可以确保始终考虑某些横切问题，例如安全性、隐私性和可观察性。这些通常是相对较短的部分，解释设计如何影响问题以及如何解决问题。团队应该对他们的案例中的这些问题进行标准化。

由于其重要性，Google 项目需要有专门的隐私设计文档，并且有针对隐私和安全的专门审查。虽然审查只需要在项目启动时完成，但最佳实践是尽早与隐私和安全团队合作，以确保设计从一开始就考虑到它们。如果有针对这些主题的专用文档，中心设计文档当然可以参考它们而不是详细介绍。

设计文档的长度

设计文档应该足够详细，但又足够短，以便忙碌的人能够真正阅读。较大项目的最佳位置似乎是 10-20 页左右。如果您超出了这个范围，那么将问题分解为更易于管理的子问题可能是有意义的。还应该指出的是，写一篇 1-3 页的“迷你设计文档”是绝对可能的。这对于敏捷项目中的渐进式改进或子任务特别有帮助。您仍然执行与较长文档相同的所有步骤，只需使内容更加简洁并专注于有限的问题集。

什么时候不应该写设计文档。

编写设计文档是一项开销。是否编写设计文档的决定归结为核心权衡，即决定围绕设计、文档、高级评审等达成组织共识的好处是否超过创建文档的额外工作。该决策的核心在于设计问题的解决方案是否不明确——因为问题复杂性或解决方案复杂性，或两者兼而有之。如果不是，那么完成编写文档的过程就没有什么价值。

可能不需要文档的一个明确指标是设计文档，它们实际上是实施手册。如果文档基本上说“这就是我们将如何实施它”，而不进行权衡、替代方案和解释决策（或者如果解决方案如此明显以至于意味着没有权衡），那么它立即编写实际的程序可能是一个更好的主意。

最后，创建和审查设计文档的开销可能与原型设计和快速迭代不兼容。然而，大多数软件项目确实存在一系列实际已知的问题。订阅敏捷方法并不是不花时间正确解决实际已知问题的借口。此外，原型制作本身可能是设计文档创建的一部分。 “我尝试了一下，它有效”是选择设计的最佳论据之一。

设计文档生命周期

设计文档生命周期的步骤是：

  1. 创建与快速迭代
  2. 审查（可能是多轮）
  3. 实施与迭代
  4. 维护与学习

创建和快速迭代

你写文档。有时与一组合著者一起。
这个阶段很快演变成一个快速迭代的时期，其中文档与对问题空间最了解的同事（通常属于同一团队）共享，并通过他们澄清问题和建议推动文档达到第一个相对稳定的版本。
虽然您肯定会发现工程师甚至团队更喜欢版本控制和代码审查工具来创建文档，但 Google 的绝大多数设计文档都是在 Google Docs 中创建的，并大量使用其协作功能。

审查

在审查阶段，设计文档会与比原始作者和密切合作者更广泛的受众共享。评论可以增加很多价值，但它们也是一个危险的开销陷阱，所以要明智地对待它们。

审查可以采取多种形式：更轻量级的版本只是将文档发送到（更广泛的）团队列表，让人们有机会查看。然后讨论主要发生在文档中的评论线程中。评审的重度部分是正式的设计评审会议，在会议上，作者向通常非常资深的工程受众展示文档（通常通过专门的演示文稿）。 Google 的许多团队都为此安排了定期会议，工程师可以报名参加审查。当然，等待此类会议的召开会大大减慢开发过程。工程师可以通过直接寻求最重要的反馈而不是阻碍更广泛审查的进展来缓解这种情况。

当谷歌还是一家规模较小的公司时，习惯上将设计发送到一个中央邮件列表，高级工程师会在闲暇时对其进行审查。这很可能是为您的公司处理事务的好方法。一个好处是它确实在整个公司建立了一种相对统一的软件设计文化。但随着公司规模扩大到更大的工程团队，维持集中式方法变得不可行。

此类评审的主要价值在于，它们为将组织的综合经验纳入设计提供了机会。最一致的是，确保设计考虑到可观察性、安全性和隐私等跨领域问题是在审查阶段可以确保的。审查的主要价值不在于问题本身被发现，而在于这种情况发生在开发生命周期的相对早期，此时进行更改的成本仍然相对较低。

实施和迭代

当事情进展到足以确信进一步的审查不太可能需要对设计进行重大更改时，就可以开始实施了。当计划与现实发生冲突时，不可避免地会出现缺陷、未解决的需求或被证明是错误的有根据的猜测，并需要更改设计。在这种情况下强烈建议更新设计文档。根据经验：如果设计的系统尚未发布，那么一定要更新文档。在实践中，我们人类不擅长更新文档，并且由于其他实际原因，更改通常被隔离到新文档中。这最终会导致一个更类似于美国宪法的国家，其中包含一系列修正案，而不是一份一致的文件。原始文档中此类修订的链接对于那些试图通过设计文档考古学来了解其目标系统的可怜的未来维护程序员来说非常有帮助。

维护与学习

当谷歌工程师面对一个他们以前从未接触过的系统时，他们的第一个问题通常是“设计文档在哪里？”。虽然设计文档与所有文档一样，随着时间的推移往往会与现实脱节，但它们仍然经常提供最容易访问的切入点，以了解指导系统创建的思想。

作为作者，请你自己帮自己一个忙，在一两年后重新阅读自己写的设计文档。你做对了什么？你做错了什么？今天你会做什么来做出不同的决定？回答这些问题是作为工程师进步并随着时间的推移提高软件设计技能的好方法。

结论

设计文档是获得清晰度并就解决软件项目中最困难的问题达成共识的好方法。他们省钱，因为他们避免陷入无法实现项目目标的编码兔子洞，并且可以通过预先调查来避免；而且它们需要花钱，因为创建和审查需要时间。因此，请为您的项目明智地选择。
在考虑编写设计文档时，请考虑以下几点：

  1. 您不确定软件设计是否正确，花费前期时间来获得确定性是否有意义？
  2. 与此相关的是，让可能无法审查每个代码更改的高级工程师参与设计是否会有帮助？
  3. 软件设计是否含糊甚至有争议，以至于围绕它达成组织共识是有价值的？
  4. 我的团队有时会忘记在设计中考虑隐私、安全、日志记录或其他跨领域问题吗？
  5. 是否强烈需要提供有关组织中遗留系统设计的高级见解的文档？
    如果您对其中 3 个或更多问题的回答是肯定的，那么设计文档可能是开始您的下一个软件项目的好方法。
